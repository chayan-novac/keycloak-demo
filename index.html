The Problem We're Solving

"Every team builds auth differently. Every app has its own users, sessions, and security gaps."

Pain Points Without Central IAM

  • Duplicate login systems across 10+ internal apps
  • No Single Sign-On — users log in repeatedly
  • MFA bolted on per-app, inconsistently
  • No audit trail for who accessed what, when
  • Custom auth code = custom vulnerabilities
  • Zero visibility into fraud or suspicious logins

The Answer

Keycloak — Open Source, Enterprise-Grade IAM

A unified identity platform that solves all standard authentication concerns out of the box, while remaining fully extensible for custom business logic.

Feature 1: Two Authentication Flows

Direct Grant Flow (Legacy / ATM)

Client → POST /token
  username=agent.platinum
  password=agent123
  banking_pin=2468
  grant_type=password

Keycloak → access_token (JWT)
  • Single HTTP call, no browser required
  • Custom PIN authenticator validates 4-digit PIN
  • Used for ATM terminals, POS systems
  • No SSO session created (API-only token)

Browser Flow (Web Apps / SSO)

Client → GET /auth?client_id=agent...
Browser → Keycloak Login Page
User → Enters credentials
Keycloak → Redirect with auth_code
Client → POST /token (+ PKCE)
Keycloak → token + SSO cookies
  • Full PKCE protection (SHA-256 challenge)
  • SSO session cookies set in browser
  • MFA integrates seamlessly
  • Credentials never exposed to the client app

Feature 2: Custom PIN Authenticator (SPI)

File: src/main/java/com/finsecure/auth/FinTechPinAuthenticator.java

What it does:

  • Plugs into Keycloak's flow as a custom Authenticator SPI
  • Reads banking_pin from the HTTP request form parameters
  • Validates against user's banking_pin attribute
  • Sets session notes (pin_validated) on success
  • Returns structured JSON error on failure

Why this matters: Legacy banking systems (ATMs, POS) can integrate with modern IAM without replacing their auth model.

Live Demo:

# Success
curl -X POST http://localhost:8082/realms/FinTechRealm/protocol/openid-connect/token \
  -d "username=agent.platinum" \
  -d "password=agent123" \
  -d "banking_pin=2468" \
  -d "grant_type=password" \
  -d "client_id=agent-portal"

# Failure — wrong PIN
curl -X POST ... -d "banking_pin=9999"
# → {"error": "Invalid banking PIN"}

Feature 3: Custom Token Claims (Protocol Mappers)

Three custom OIDC Protocol Mappers inject business context directly into JWT tokens. No downstream DB calls needed.

1. AgentTierMapper

Reads customers_acquired → calc tier.

  "customers_acquired": 250,
  "agent_incentives": {
    "commission_rate_pct": 5.0,
    "monthly_bonus": 2000,
    "next_tier": "MAX_TIER"
  }
}

2. AccountTierMapper

Reads account_balance → calc tier.

Feature 5: Fraud Detection Event Listener (SPI)

File: src/main/java/com/finsecure/event/FraudDetectionEventListener.java

Keycloak fires events on every auth action. This SPI intercepts them all to detect anomalies.

Events Monitored:

EventAction Taken by SPI
LOGINUpdate last_login, last_ip, reset counters
LOGIN_ERRORIncrement fails, auto-disable after 5
LOGOUTRecord last_logout_at
UPDATE_PASSWORDDetect IP change, flag for alert
Admin EventsAlert on ROLE and USER changes

Fraud Patterns Detected:

  • Login from new IP address
  • Rapid re-login (< 60 seconds between logins)
  • Login between 2 AM – 5 AM
  • Multiple failed attempts → auto account lock

Production Integration:

Easily routable to SIEM, Elasticsearch, or SOC notifications via Kafka integration.

Feature 6: Transaction Limit Auth Policy (SPI)

File: src/main/java/com/finsecure/policy/TransactionLimitPolicy.java

A custom Keycloak Authorization Policy that enforces transaction limits at the IAM layer.

Decision Logic:

1. Extract transaction_amount from context
2. Verify KYC status = "VERIFIED" → deny if not
3. Calculate account tier from account_balance
4. Compare amount against tier limit
5. Grant or Deny access

Why this matters: Authorization policy lives in the central IAM layer — not scattered across microservices. One policy, enforced everywhere.

Tier Limits Enforced:

TierDaily Transaction Limit
STANDARD$5,000
SILVER$25,000
GOLD$100,000
PLATINUM$500,000

Feature 7: Single Sign-On (SSO)

Demo Portal: demo-apps/web/sso-demo.html

How it works:

Step 1: User logs into Agent Portal
        → Keycloak sets SSO cookie in browser

Step 2: User opens Customer Portal
        → Browser sends SSO cookie
        → Keycloak recognizes session 
        → Issues token (NO LOGIN PROMPT)

Step 3: User logs out from any app
        → Keycloak invalidates SSO session
        → All apps lose access simultaneously

Session Configuration:

  • SSO Session Idle: 30 minutes
  • SSO Session Max: 10 hours
  • Access Token Lifetime: 5 minutes
  • Protocol: Auth Code + PKCE
Live Test:
  1. Login at /agent-portal.html
  2. Open /customer-app.html
  3. Click Login → Instant Access

Feature 8: Multi-Factor Authentication (MFA)

Protocol: TOTP (RFC 6238) — 6-digit, 30-second window

MFA + SSO Behavior:

Login App 1 → Password → OTP ✓
Open App 2  → Auto-auth (SSO) — NO OTP
Open App 3  → Auto-auth (SSO) — NO OTP
Logout      → All sessions terminated
Next login  → OTP required again

MFA is validated once per SSO session — not per application.

Setup Flow:

  1. Login to any portal
  2. Navigate to Keycloak Account Console
  3. Security → Two-factor Authentication → Scan QR

Compatible with Google Authenticator, Authy, Microsoft Authenticator.

Brute Force Protection:

  • 5 failed attempts → account temporarily locked
  • Configurable lockout duration
  • All failures logged by Fraud Detection listener

Quick Start

# One command to run everything
cd keycloak-demo
chmod +x scripts/setup-complete-demo.sh
./scripts/setup-complete-demo.sh

What it does:

  1. Builds the custom SPI JAR (mvn package)
  2. Builds the Docker image with JAR pre-loaded
  3. Starts KC 26 + PostgreSQL via Docker Compose
  4. Runs setup-realm.py (creates clients, users)
  5. Starts the static web server for demo portals
