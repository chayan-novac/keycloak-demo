The Problem We're Solving

"Every team builds auth differently. Every app has its own users, sessions, and security gaps."

Pain Points Without Central IAM

  • Duplicate login systems across 10+ internal apps
  • No Single Sign-On — users log in repeatedly
  • MFA bolted on per-app, inconsistently
  • No audit trail for who accessed what, when
  • Custom auth code = custom vulnerabilities
  • Zero visibility into fraud or suspicious logins

The Solution: Keycloak

A battle-tested, open-source IAM platform used by enterprises worldwide. One platform. All auth concerns solved.

  • Trust: Used by Red Hat, Bosch, Airbus
  • Modern: CNCF-listed, actively maintained
  • Protocols: OpenID Connect, OAuth 2.0, SAML 2.0
  • Extensible: Fully customizable via Java SPI

Core Concept: Realm

A Realm is an isolated security domain. Think of it as a tenant.

What it gives you

  • Separate user stores, clients, roles, and policies per realm
  • master realm for admin configuration
  • Create dedicated realms per product/environment
  • Example: partnerone-prod, partnerone-dev

Why it matters

  • Zero cross-contamination between environments
  • Each realm has its own token signing keys, session policies, and UI branding
  • Multi-tenancy out of the box — no custom code needed

Clients & Application Integration

A Client is any application that delegates authentication to Keycloak.
Type Use Case
public Mobile apps, SPAs (no client secret embedded)
confidential Backend services, APIs (authenticates with client secret)
bearer-only Microservices that only validate tokens, never initiate login

Supported Flows Per Client:

  • Authorization Code + PKCE: Secure flow for mobile/web browsers
  • Client Credentials: Machine-to-machine / service-to-service
  • Direct Grant: Username + password → token (Used in our MPIN flow)
  • Device Authorization: For IoT and Smart TVs

Protocol & Audience Mappers

Protocol Mappers control what data gets baked into the JWT token.

Out-of-the-box Mappers

  • User attribute → token claim
  • Role → claim
  • Group membership → claim
  • Hardcoded claim values

Audience Mapper

  • Adds aud claim so downstream services can verify intent
  • Prevents token reuse across different services
  • Example: Token for mobile-app cannot be accepted by admin-service

Our Project Uses This For: Embedding custom_user_id, mobile_number, device_id, and date_of_birth into the JWT claims for frictionless downstream service consumption.

Feature: Two Authentication Flows

1. Direct Grant Flow (Mobile/API)

Client → POST /token
  username=user@example.com
  mpin=1234
  grant_type=password
  client_id=mobile-app

Keycloak → access_token (JWT)
  • Single HTTP call, no browser
  • Custom MPIN authenticator validates 4-digit PIN
  • Tracks failed attempts in DB, auto-locks
  • No SSO session created (API-only)

2. Browser Flow (Web/SSO)

Client → GET /auth?client_id=...
Browser → Keycloak Login Page
User → Enters credentials
Keycloak → Redirect with auth_code
Client → POST /token (code + PKCE)
Keycloak → token + SSO cookies
  • Full PKCE protection (SHA-256)
  • SSO session cookies set
  • Credentials never exposed to app

Our "MPIN with Lockout" Flow: Replaces standard password auth. Validates MPIN, locks after MPIN_MAX_ATTEMPTS (3) for MPIN_LOCKOUT_MINUTES (15m).

Feature: Custom SPI Interfaces

Keycloak's SPI framework lets you plug in custom Java logic at every layer.

1. MpinAuthenticator

  • Plugs into Direct Grant flow
  • Validates 4-digit MPIN
  • Tracks fails in user_mpin PostgreSQL table
  • Auto-locks & auto-unlocks based on env vars

2. Security Questions Credential

  • New credential type: security-questions
  • Stores 3 Q&A pairs as JSON securely
  • Used as identity verification factor in Forgot MPIN flow

3. MpinPolicyProvider

  • Enforces PIN rules at the identity layer
  • Rejects: non-4-digit, letters, repeated (1111), sequential (1234)
  • Works across all flows natively

4. KafkaEventListenerProvider

  • Hooks into REGISTER & UPDATE_PASSWORD
  • Publishes structured onboarding payloads to Kafka/Azure
  • Includes custom metadata, questions, and MPIN hash

Feature: Multi-Factor Auth (MFA)

Built-in MFA Options

  • TOTP (Google Authenticator)
  • HOTP (Counter-based OTP)
  • WebAuthn / FIDO2 (Passkeys, Hardware keys)

Our Custom MFA (SMS OTP)

  • Sent via SMSService to mobile
  • Stored in Redis with TTL & device binding
  • IP Rate Limiting (OTP_MAX_IP_REQ)
  • Mobile Rate Limiting
  • Soft lock (session) & Hard lock (24h)

Dev Tooling: Whitelisted numbers bypass OTP for automated testing via mobile-whitelist.txt.

Feature: Single Sign-On (SSO)

SSO means authenticate once, access everything.

How Keycloak SSO works:

  1. User logs into App A → Keycloak creates a session cookie
  2. User navigates to App B → Keycloak detects existing session
  3. App B gets a token without the user re-entering credentials

Session Management Features:

  • Configurable SSO session idle / max timeouts per realm
  • Admin can view and kill active sessions in real time
  • backchannel-logout propagates logout to all connected applications
  • Cross-realm SSO: Identity brokering allows enterprise multi-product SSO

Feature: RBAC & Groups

Role Types

Realm rolesAvailable across all clients
Client rolesScoped to specific app
Composite rolesRoles containing other roles

Groups

  • Hierarchical (org > dept > team)
  • Assign roles to groups; users inherit them instantly
  • Group memberships map to JWT claims

Example Mapping:

Group: /finance/analysts
  → Realm Role: view-reports
  → Client Role: reporting-app:read-only

Any user added to this group gets both roles in their token automatically.

Feature: Custom Themes

Every screen Keycloak renders can be fully branded.
  • Themeable surfaces: Login, registration, error pages, emails, account portal.
  • Types: login, account, admin, email, welcome.
  • How: Drop a theme folder into /opt/keycloak/themes/.
  • Result: Users see your brand, not Keycloak's. Zero custom auth server code needed for UI.

Identity Brokering & Federation

Keycloak can act as a broker — delegating auth to external IdPs.
  • Supported IdPs: Google, Apple, Microsoft, GitHub, any SAML/OIDC provider.
  • User Federation: Connect to LDAP / Active Directory. Migrate legacy users lazily on first login.
  • Our Project Context: Users are created natively via our Registration SPI. Extended attributes map to PostgreSQL and Redis for sub-millisecond lookups.

Infrastructure & Supporting Services

DatabaseConnectionPool

  • HikariCP PostgreSQL pool
  • Singleton, thread-safe
  • Used across all REST providers & SPIs

RedisOtpService

  • Jedis pool-based client
  • Rate limiting (per-IP, per-mobile)
  • Soft/Hard lockouts & Device Binding

KafkaProducerService

  • High-performance async producer
  • Jackson + Afterburner for JSON
  • Dual-mode: Local Kafka or Azure Event Hubs

UserExistenceService

  • 3-layer lookup: Redis → DB → Keycloak
  • Sub-millisecond response
  • Cache warming utilities included
