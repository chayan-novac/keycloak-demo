The Challenge

Why Centralized IAM Matters

"Every team builds auth differently. Every app has its own users, sessions, and security gaps."

Duplicate Login Systems

10+ internal apps each with their own authentication implementation

No Single Sign-On

Users log in repeatedly across different applications

Inconsistent MFA

MFA bolted on per-app with varying security levels

No Audit Trail

Zero visibility into who accessed what and when

Custom Vulnerabilities

Custom auth code means custom security gaps

Fraud Detection Gaps

No centralized monitoring for suspicious login patterns

The Solution: Keycloak — Open Source, Enterprise-Grade IAM. Used by Red Hat, Bosch, Airbus, and thousands of enterprises. CNCF-listed project with active maintenance.

Core Features

Comprehensive IAM features implemented in this demonstration.

01 Two Authentication Flows

Direct Grant Flow

For legacy systems, ATMs, server-to-server

  • Single HTTP call, no browser
  • Custom PIN authenticator
  • No SSO session created
  • API-only token response

Browser Flow

For web apps with full SSO support

  • Full PKCE protection (SHA-256)
  • SSO session cookies set
  • MFA integrates seamlessly
  • Credentials never exposed
Direct Grant Request
POST /realms/FinTechRealm/protocol/openid-connect/token
Content-Type: application/x-www-form-urlencoded

username=agent.platinum
password=agent123
banking_pin=2468
grant_type=password
client_id=agent-portal

02 Custom PIN Authenticator (SPI)

A custom Authenticator SPI that validates 4-digit banking PINs in the authentication flow.

  • ✓

    PIN Validation

    Reads banking_pin from HTTP form parameters and validates against user attributes

  • ✓

    Session Notes

    Sets pin_validated and validation_timestamp on success

  • ✓

    Structured Errors

    Returns JSON error responses on validation failure

Why it matters: Legacy banking systems (ATMs, POS) can integrate with modern IAM without replacing their auth model.

03 Custom Token Claims (Protocol Mappers)

Three custom OIDC Protocol Mappers inject business context directly into JWT tokens.

AgentTierMapper

Reads customers_acquired attribute, calculates tier, and injects into token:

{ "agent_tier": "PLATINUM", "customers_acquired": 250, "agent_incentives": { "commission_rate_pct": 5.0, "monthly_bonus_usd": 2000, "next_tier": "MAX_TIER", "benefits": ["5% commission", "$2000/mo bonus", "Priority leads"] } }

AccountTierMapper

Reads account_balance and calculates tier with configurable thresholds:

Tier Balance Threshold Benefits
PLATINUM $100,000+ Priority support, Zero fees, Dedicated manager
GOLD $50,000+ Premium support, Reduced fees
SILVER $10,000+ Extended support hours
STANDARD < $10,000 Basic support

TransactionRiskScoreMapper

Multi-factor risk scoring injected into every token:

Factor Points
Account age < 30 days+20
Failed login attempts+5 per attempt (max 25)
KYC not verified+30
Login from new IP+15

05 Fraud Detection Event Listener (SPI)

Keycloak fires events on every auth action. This SPI intercepts and processes them all.

Event Action Taken
LOGINUpdate last_login_at, last_known_ip, reset failed counter
LOGIN_ERRORIncrement failed_login_attempts, auto-disable after 5 failures
LOGOUTRecord last_logout_at
UPDATE_PASSWORDDetect password change from new IP, alert
VERIFY_EMAILRecord email_verified_at
Admin EventsAlert on ROLE and USER resource changes

Fraud Patterns Detected

New IP Login

Detect login from previously unknown IP address

Rapid Re-login

< 60 seconds between logins triggers alert

Off-Hours Access

Login between 2 AM – 5 AM flagged

Auto Lock

Multiple failed attempts triggers account lock

07 Single Sign-On (SSO)

Authenticate once, access everything. SSO session cookies enable seamless cross-application access.

1

Initial Login

User logs into Agent Portal → Keycloak sets SSO session cookie in browser

2

Second App Access

User opens Customer Portal → Browser sends SSO cookie → Keycloak recognizes session → Skips login

3

Unified Logout

User logs out from any app → Keycloak invalidates SSO session → All apps lose sessions simultaneously

Session Configuration

Setting Value
SSO Session Idle30 minutes
SSO Session Max10 hours
Access Token Lifetime5 minutes
ProtocolAuthorization Code + PKCE

08 Multi-Factor Authentication (MFA)

Protocol: TOTP (RFC 6238) — 6-digit codes, 30-second window

MFA + SSO Behavior: MFA is validated once per SSO session — not per application. Login to App 1 with OTP, then App 2 and App 3 auto-authenticate without additional OTP prompts.

Setup Flow

  1. Login to any portal
  2. Navigate to http://localhost:8082/realms/FinTechRealm/account
  3. Security → Two-factor Authentication → Scan QR
  4. Compatible with Google Authenticator, Authy, Microsoft Authenticator

Built-in MFA Options

  • ✓

    TOTP

    Time-based OTP (Google Authenticator, Authy) — RFC 6238

  • ✓

    WebAuthn / FIDO2

    Passkeys, hardware security keys

  • ✓

    Brute Force Protection

    5 failed attempts → account temporarily locked

Demonstration

Live Demo Scenarios

Step-by-step scenarios to demonstrate key IAM capabilities.

A PIN Authentication

  1. Open Agent Portal → Direct Grant tab
  2. Enter: agent.platinum / agent123 / PIN: 2468
  3. Click Authenticate
  4. Show decoded JWT → agent_tier: PLATINUM, commission: 5%
  5. Enter wrong PIN → show rejection

B SSO in Action

  1. Open agent-portal.html → Browser Flow → Login
  2. Open customer-app.html in same browser
  3. Click Login → instant access (no credentials)
  4. Logout from customer portal
  5. Try agent portal → session terminated

C Agent Tier Progression

  1. Authenticate as agent.standard (5 customers, 1% commission)
  2. Click "Record Customer Acquisition" × 5
  3. Watch tier change: STANDARD → SILVER
  4. New commission: 2%, new bonus: $250/month

D Token Claims Inspection

  1. Authenticate as john.platinum
  2. Show decoded token:
    • account_tier: PLATINUM
    • transaction_risk_score: 5 (MINIMAL)
    • agent_incentives: {...}
  3. Compare with bob.standard → STANDARD tier, $5k limit

E Fraud Detection

  1. Enter wrong password 3× for john.platinum
  2. Open Keycloak Admin → Events → LOGIN_ERROR
  3. Show failed_login_attempts attribute on user
  4. 5th failure → account disabled automatically
Security

Security Highlights

Production-grade security patterns implemented in this demo.

Concern Implementation
Brute Force on MPIN3-attempt lockout, 15-min timeout, tracked in DB
OTP AbusePer-IP + per-mobile rate limiting, device binding, hard lock after exhaustion
Token SecurityShort-lived JWTs, audience mapper, realm isolation
PAN/Mobile IntegrityDB-enforced 1:1 mapping, checked at registration
Device LimitsRule engine: max 3 users/device, max 3 devices/user with cooldown
Session SecurityTime-limited session tokens, step-gated progression
Credential StorageMPIN via Keycloak credential manager (hashed), SHA-256 in custom table
Event AuditAll auth events captured, published to Kafka for SIEM/audit trail
